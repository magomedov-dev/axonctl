# Концепции

[English](concepts.md) · **Русский**

## Устройство без состояния

Агент на устройстве не хранит состояние между вызовами. Каждый дамп начинается со
свежего корня; id узлов валидны только в пределах одного дампа и никогда не
возвращаются на устройство. **Всё** состояние живёт в контроллере:

- Поиск элементов, навигация по дереву (родители/дети/потомки), частичное
  совпадение и regex — всё на ПК поверх дампа (см. [`Selector`][axonctl.Selector],
  [`UiTree`][axonctl.UiTree]).
- Ожидания живут на ПК и событийны.
- Ретраи (например, при `STALE`) решает контроллер; устройство не ретраит.

Это держит устройство простым и быстрым, а всю логику — там, где её можно
протестировать.

## Событийные ожидания, не поллинг

Ожидание подписывается на поток событий `screenChanged` агента, делает базовый
дамп и **передампливает только** когда экран реально изменился. Никакого цикла
опроса. Счётчик `screen` монотонен, поэтому «что-то изменилось» — дешёвый сигнал
перепроверить.

```python
# Резолвится, как только элемент появился — без busy-wait.
node = await device.wait_for(Selector.id("com.app:id/ok"), timeout=10)
```

Транзиентное «нет активного окна» в момент перехода (например, при запуске
приложения) терпится: ожидание просто продолжает ждать, а не падает.

Библиотека включает поток событий агента **автоматически при подключении**
устройства (и заново — после реконнекта), поэтому события идут с самого начала —
`setEventStream` вызывать вручную не нужно. Поскольку поток включён рано,
`wait_toast` умеет вернуть тост, всплывший в коротком окне *прямо перед* вызовом,
закрывая гонку, когда фидбэк формы («Неверный пароль») появляется между возвратом
действия и подпиской ожидания.

## Группы и теги

Теги **статические**, задаются в конфиге (`serial -> tags`). Они не читаются с
устройства и не меняются на лету; индекс тегов обновляется только при
attach/detach. [`run`][axonctl.FleetController.run] выбирает цель по имени группы,
явному списку серийников, предикату по тегам или весь парк.

## Модель исполнения — один loop, много устройств

Это важнейшее, что нужно понять до написания сценариев.

Каждый сценарий на каждом устройстве выполняется как **конкурентная asyncio-таска
в одном потоке и одном event loop** — не потоки, не процессы. Именно это позволяет
одному процессу управлять сотнями устройств: нагрузка почти целиком I/O (ожидание
сети и телефонов), а оно не занимает CPU. Пока устройство A ждёт ответ по сокету,
loop обслуживает B, C, D.

Следствие: **одна ошибка в твоём сценарии может застопорить весь парк.** Правило —
внутри сценария всё должно быть либо *быстрым* (миллисекунды), либо *await-ным*.

Три случая обработки данных в сценарии:

1. **Быстрая работа (мс):** разбор, условия, выбор следующего действия. Ничего не
   надо — миллисекундный всплеск CPU незаметен для других устройств (они всё равно
   ждут свой I/O).
2. **Работа, ждущая внешний I/O** (свой API, БД, скачивание, пауза): async-
   инструменты — `httpx`/`aiohttp` вместо `requests`, async-драйвер БД,
   `await device.sleep(...)` / `await asyncio.sleep(...)` вместо `time.sleep(...)`.
   Тогда ожидание перекрывается и парк не стоит.
3. **Тяжёлый CPU секундами** (картинки, ML, крипта): вынести с loop через
   `await loop.run_in_executor(pool, fn, data)` — `ProcessPoolExecutor` для CPU,
   `ThreadPoolExecutor` для блокирующего I/O без async-версии.

### Антипаттерн: блокировка loop

Любой блокирующий вызов в сценарии — `time.sleep`, синхронный `requests`,
блокирующий драйвер БД, секундный CPU-цикл — **замораживает event loop и
останавливает все остальные устройства разом.** Это причина №1 «мистических
тормозов парка».

| Блокирующий вызов        | Async-замена                                    |
|--------------------------|-------------------------------------------------|
| `time.sleep(n)`          | `await device.sleep(n)` / `await asyncio.sleep(n)` |
| `requests.get(...)`      | `await httpx.AsyncClient().get(...)` / `aiohttp` |
| синхронный драйвер БД    | async-драйвер (`asyncpg`, `aiosqlite`, ...)     |
| `open(...).read()` (большой) | `await loop.run_in_executor(pool, ...)` / `aiofiles` |
| CPU-bound `fn(data)`     | `await loop.run_in_executor(process_pool, fn, data)` |
| `subprocess.run(...)`    | `await asyncio.create_subprocess_exec(...)`     |

Сам [`Device`][axonctl.Device] уже неблокирующий; ответственность за
неблокирующий *пользовательский* код — на авторе сценария.

Практические паттерны — в [Написании сценариев](scenarios.md).
